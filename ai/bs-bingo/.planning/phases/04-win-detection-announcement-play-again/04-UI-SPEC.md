# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4 (Win Detection, Announcement & Play-Again). Generated by gsd-ui-researcher. Inherits Phase 1/2/3 design system; extends it with the `ended` phase surface (EndScreen), the win-line mini-grid icon, and the host-only play-again CTA. No new tokens; no new dependencies beyond `canvas-confetti`.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Tailwind v4 + `@theme` tokens, no component library) |
| Preset | not applicable (shadcn is React-only; stack is SvelteKit + Svelte 5) |
| Component library | First-party Svelte components under `src/lib/components/` — Phase 4 adds `EndScreen`, `WinLineIcon` |
| Icon library | `lucide-svelte` ^0.454.0 — one new icon used: `party-popper` on the winner's EndScreen heading (optional, accent-colored). No icon required on non-winner view. |
| Font | Inter Variable (body/label/heading) + Space Grotesk Variable (display) — unchanged |

Source: Phase 1/2/3 UI-SPEC (inherited), `src/app.css` @theme tokens (verified 2026-04-18), Phase 4 CONTEXT.md D-01 through D-14.

Phase 4 surface scope: `<EndScreen />` component rendered inside existing `/room/[code]` route when `state.phase === 'ended'` (CONTEXT D-01, D-12). The same page continues to show the room header (code, share link) unchanged; the below-header region swaps from `<Board />` to `<EndScreen />`. After the host triggers play-again, the region swaps back to `<Lobby />` and the roster re-renders word-pool UI.

**New dependency:** `canvas-confetti` ^1.9.3 (~10KB gzipped per CONTEXT D-04). Vanilla browser library, no component-registry payload — falls outside the registry safety gate.

Unchanged from Phase 1/2/3. One new layout constant for the EndScreen hero composition.

| Token | Value | Usage |
|-------|-------|-------|

| sm | 8px | Cell-to-cell gap on frozen winner board (inherited), 8px internal spacing on WinLineIcon |
| md | 16px | Default section padding, gap between EndScreen header block and mini-grid icon |
| lg | 24px | Vertical rhythm between EndScreen blocks (heading → winner line → mini-grid → CTA) |

**Exceptions:**
- **44px minimum tap target** on "Start New Game" CTA (inherited Phase 1 rule). Host-only control, primary button sizing.
- **Container max-width: 640px** on `/room/[code]` (inherited from Phase 1). EndScreen content centers within this container.
- **Mini-grid icon size: 64px × 64px** at all grid tiers (3×3 / 4×4 / 5×5). Internal cells scale to fit. Non-negotiable — larger would compete with the frozen-board view; smaller would lose line recognizability.
- **Confetti canvas: fixed, inset-0, `pointer-events: none`, `z-index: 50`.** Full-viewport overlay so particles can fall from above into the screen on the winner's view. Disabled / not mounted on non-winner clients.

## Copywriting Contract

Voice inherited: fun, slightly irreverent, not corporate. Second person. Short sentences. Celebration copy leans into "you did the thing" rather than prize-ceremony formality.

| Element | Copy |
|---------|------|
| Winner — Display wordmark | `BINGO!` |
| Winner — name line (below wordmark) | `{YourDisplayName}` |
| Winner — subline (below name) | `You called it. {LineType} {LineIndex}.` (e.g. `You called it. Row 3.`, `You called it. Top-left diagonal.`) |
| Non-winner — heading | `{WinnerDisplayName} got Bingo!` |
| Non-winner — subline (under mini-grid icon) | `{LineType} {LineIndex} completed.` (e.g. `Row 3 completed.`, `Top-left diagonal completed.`) |
| Non-winner — closing line (below subline, optional) | `Nice try. One more round?` |
| Play-again CTA (host) | `Start new game` |
| Play-again helper (host, below CTA) | `Word pool and players are kept. You can tweak the pool before starting.` |
| Play-again placeholder (non-host) | `Waiting for the host to start a new game.` (Body 16px, `#A1A1AA`) |
| Win-line label mapping — row | `Row {1-indexed number}` (e.g. `Row 1`, `Row 3`) |
| Win-line label mapping — column | `Column {1-indexed number}` (e.g. `Column 2`) |
| Win-line label mapping — diagonal (index 0, main) | `Top-left diagonal` |
| Win-line label mapping — diagonal (index 1, anti) | `Top-right diagonal` |
| Game-reset transition — no copy | (silent — on `gameReset` the EndScreen unmounts and lobby re-renders; no intermediary flash/toast) |
| Error: winner data incomplete (defensive) | `Someone got Bingo, but we lost the details. Ready for another?` (Body 16px, shown only if `winDeclared` arrived with missing fields — should be unreachable) |
| Aria-label — "Start new game" button | `Start a new game with the same lobby.` |
| Aria-label — WinLineIcon (non-winner) | `Winning line: {LineType} {LineIndex}.` (e.g. `Winning line: Row 3.`) |
| Aria-label — "BINGO!" wordmark (winner) | `Bingo. You won with {LineType} {LineIndex}.` |
| Aria-live — EndScreen on mount (non-winner) | `{WinnerDisplayName} got Bingo with {LineType} {LineIndex}.` (polite) |
| Aria-live — EndScreen on mount (winner) | `Bingo. You won with {LineType} {LineIndex}.` (polite) |

**Line-type string source of truth:** a single `formatWinLine(line: { type, index }): string` helper in `src/lib/utils/winLine.ts` returns the human-readable string used everywhere — button aria-labels, sublines, aria-live announcements. Index translation: rows/columns are 1-indexed in UI, 0-indexed in protocol; diagonals map from `index: 0 | 1` to `Top-left diagonal | Top-right diagonal`.

**Destructive actions in Phase 4:** **None.**
- "Start new game" is not destructive from the player's perspective — the word pool and roster are retained (CONTEXT D-10). It only clears boards and marks, which have served their purpose. No confirmation modal.
- Leaving the room via browser back / closing tab is out of scope for Phase 4 (Phase 5 owns reconnect and resume).
- There is no "dismiss EndScreen" affordance — the screen persists until the host triggers a new game. This is by design per CONTEXT D-08.

**Empty state:** Not applicable in Phase 4. EndScreen only renders when `phase === 'ended'`, which only occurs after a valid `winDeclared` broadcast. There is no "no winner yet" EndScreen variant; that state is covered by Phase 3's Board.

**Error state taxonomy:**
- **Incomplete `winDeclared` payload (defensive):** render the defensive copy above inside the EndScreen container. Preserve the Start New Game CTA so the host can recover. Fall back to "Someone" for missing winnerName, and omit the win-line subline if the line data is missing.
- **`startNewGame` fails silently (non-host tries, DO drops message per CONTEXT D-13):** no error surfaced; the button is simply not rendered to non-hosts, so this state is unreachable via the UI.
- **Disconnect during EndScreen:** inherited Phase 1 `Reconnecting…` banner overlays the EndScreen. The EndScreen state is preserved locally (the winner + winning line + frozen board are already in client state). Phase 5 owns deeper reconnect UX; Phase 4 leaves the EndScreen visible behind the reconnecting banner.
- **`gameReset` arrives while EndScreen is visible:** the client store clears board/marks/winner state and flips `phase → 'lobby'` (CONTEXT D-10). The EndScreen unmounts cleanly via the existing `{:else if phase === 'ended'}` branch; the Phase 2 lobby surface mounts in its place. No toast, no transition copy — see Motion for the crossfade.

## Interaction Contracts

Inherits Phase 1/2/3 interaction states. Phase 4 additions:

| Element | Default | Hover | Active / Pressed | Focus-visible | Disabled |
|---------|---------|-------|------------------|---------------|----------|
| "Start new game" CTA (host) | `bg-accent text-ink-inverse` (inherits Phase 1 primary button) | 8% lighter accent | translateY(1px), 60ms | 2px outline `#A1A1AA` offset 2px | not rendered to non-hosts (no disabled state) |
| "Start new game" helper (non-host) | Body 16px `#A1A1AA` text | — | — | — | not interactive |
| Frozen board cell (winner view, any state) | inherited from Phase 3 BoardCell colors | no hover (cells are `pointer-events: none` in EndScreen per D-08) | — | not focusable (`tabindex="-1"`) | permanent disabled (freeze) |
| WinLineIcon | Static 64px card; `cursor: default` | — | — | — | non-interactive |
| Confetti canvas | Full-viewport overlay on winner view; `pointer-events: none` | — | — | — | unmounted on non-winner clients |

**Board freeze behavior (CONTEXT D-08, D-03):**
- On `winDeclared`, the `<Board />` component unmounts and a **frozen snapshot** of the same cells renders inside `<EndScreen />` for the winner only.
- The snapshot uses the exact same `BoardCell` component, but the parent wraps the grid in a `<div style="pointer-events: none;">` so no marks can register.
- Win-line cells receive the ring glow via a `data-win-line="true"` attribute (so the existing `BoardCell.svelte` gets one additional CSS rule scoped to that attribute — no new component).
- Non-winners do **not** see any board on the EndScreen — only the mini-grid icon + name + copy. This preserves BOAR-03 (private layouts) through the end of the game.

**"Start new game" flow (CONTEXT D-09, D-10, D-11):**
- Host clicks → client sends `startNewGame` ClientMessage → no local optimistic transition (wait for server confirmation).
- On `gameReset` ServerMessage: client store clears board/marks/winner fields, flips `phase → 'lobby'`, keeps word pool + players + host role.
- EndScreen unmounts, Lobby mounts. Host lands on the same lobby view they saw pre-game, with the existing word pool editable (add/remove words, re-seed a pack, etc.).
- There is no auto-start, no countdown, no confirmation modal (CONTEXT D-11).

**Non-host post-win behavior:**
- Non-hosts see the EndScreen with the mini-grid icon, winner's name, and the "Waiting for the host to start a new game." helper text where the CTA sits on the host's view.
- When `gameReset` arrives (broadcast), non-hosts' clients flip to lobby identically — no acknowledgment UI needed on the host's side; the broadcast is the signal.

**Confetti trigger (winner only):**
- On the winning client's receipt of `winDeclared` where `winnerId === selfPlayerId`, call `confetti()` once with the Phase 4 particle config (see Motion).
- Confetti does NOT fire for non-winners (CONTEXT D-04).
- Confetti does NOT re-fire on reconnect (hydration of `winner` state without a fresh `winDeclared` does not trigger). This is handled by guarding the call on the *message handler*, not on `$effect` reactive state.

**Keyboard interaction:**
- `Tab` on the winner's EndScreen: skips the frozen board (all cells are `tabindex="-1"`), lands on the "Start new game" CTA (host) or nothing interactive (non-host).
- `Enter` / `Space` on the "Start new game" CTA triggers the flow identical to a click.
- `Esc` has no effect on the EndScreen (there is nothing to dismiss).

## Responsive Behavior

Inherits Phase 1/2/3 breakpoints. Phase 4 EndScreen layout:

| Breakpoint | Width | EndScreen layout |
|------------|-------|------------------|
| base | <640px (mobile portrait) | Single column, centered. Vertical order: "BINGO!" (winner) or "{Name} got Bingo!" (non-winner) → subline → WinLineIcon (non-winner) OR frozen board (winner) → "Start new game" CTA (host) or helper text (non-host). 16px page padding. `lg` (24px) vertical rhythm between blocks. |
| sm | ≥640px | Same single column, container max-width 640px. Display size bumps to 56px. Outer vertical padding `xl` (32px). |
| md | ≥768px | No structural change — EndScreen stays a single centered column. Frozen board and WinLineIcon remain inline in the stack. Desktop feel comes from the additional whitespace, not a two-column layout. (Claude's Discretion: avoids reflow between mobile and desktop compositions.) |

**Why no two-column on desktop:** the EndScreen is a moment, not a workspace. Keeping it single-column on desktop preserves the vertical drama (BINGO → name → board/icon → new game) and avoids a wide layout where the eye has to choose between two targets.

**Orientation:**
- Portrait mobile is the canonical layout.
- Landscape mobile: the frozen board (winner view) may push the Start New Game CTA off the first fold. Acceptable — the CTA is not required on first view; the celebration IS the first view. Scroll is fine.
- Landscape mobile (non-winner): no frozen board, so the composition fits comfortably above the fold.

**Confetti canvas:** full-viewport (`position: fixed; inset: 0`) regardless of breakpoint. `pointer-events: none` so it never intercepts taps.

## Motion & Animation

Extends Phase 1/2/3 motion contract. Phase 4 is **where celebration happens** — the restraint rules from prior phases relax specifically on the winner's EndScreen.

| Moment | Motion |
|--------|--------|
| Phase transition `playing → ended` | Crossfade: Board fades out (120ms ease-out), EndScreen fades in (150ms ease-out). Same family as Phase 3's `lobby → playing` transition. Roster stays mounted; only the below-roster region swaps. |
| "BINGO!" wordmark entrance (winner) | Scale 0.9 → 1.0 + fade-in over 240ms, `ease-out`. Starts 80ms after EndScreen fade-in completes so the transition composes as "board out → screen in → BINGO pops." |
| Winner name + subline entrance | Fade-in 180ms, starts 160ms after EndScreen mount (staggered behind the wordmark). |
| Frozen board entrance (winner view) | Fade-in 200ms with a 20ms per-row stagger (inherited Phase 3 `boardAssigned` cadence, reused for visual consistency). Starts 240ms after EndScreen mount so the BINGO wordmark has time to land. |
| Win-line ring glow on frozen cells | Pulse: `box-shadow: 0 0 0 2px #F5D547, 0 0 8px #F5D547` → `0 0 0 2px #F5D547, 0 0 16px #F5D547` → back. 1200ms cycle, `ease-in-out`, infinite. Keeps the winning line visually alive while the EndScreen persists. Collapses under reduced-motion to a static `0 0 0 2px #F5D547, 0 0 12px #F5D547`. |
| Mini-grid icon entrance (non-winner view) | Fade-in 180ms + scale 0.96 → 1.0, `ease-out`, starts 120ms after EndScreen mount. |
| WinLineIcon highlighted cells (non-winner) | Static — no motion beyond the entrance. Accent-free so it doesn't compete for attention. |
| "Start new game" CTA entrance | Fade-in 160ms, starts 360ms after EndScreen mount (last element to arrive on the stage). |
| Confetti (winner only) | One burst on `winDeclared` receipt: `confetti({ particleCount: 180, spread: 90, startVelocity: 45, ticks: 220, origin: { y: 0.25 }, colors: ['#F5D547', '#F5F5F7', '#F87171'] })`. Single call, non-repeating. Canvas unmounts after ~3s when particles settle. |
| Confetti — reduced motion | Single fire, but with `particleCount: 60`, `ticks: 100`, no `startVelocity` override (relies on library default fall). Collapses duration to ~1.5s. Canvas still unmounts after particles settle. Never disabled entirely — the burst IS the celebration; reducing it is appropriate, removing it would break the contract. |
| Phase transition `ended → lobby` (on `gameReset`) | Crossfade: EndScreen fades out (120ms), Lobby fades in (120ms). Inverse of the enter transition. Confetti canvas (if still mounted) unmounts immediately on transition start. |

**Respect `prefers-reduced-motion: reduce`:**
- Phase transitions crossfade collapses to instant swap.
- BINGO wordmark entrance collapses to instant (no scale, no fade).
- Winner name / subline / frozen board / mini-grid / CTA entrances collapse to instant (no stagger).
- Win-line ring glow collapses to static (no pulse).
- Confetti uses the reduced-motion config above — reduced, not removed.

**Why reduced-motion still fires confetti:** CONTEXT D-04 specifies confetti as a load-bearing part of the winner experience. Removing it entirely would change the contract from "you won" to "something happened." Reducing it to a smaller burst preserves the signal while respecting the user's preference.

**Modified protocol types (source: CONTEXT D-12, D-13, D-14):**
```ts
// src/lib/protocol/messages.ts additions
type WinningLine = {
  type: 'row' | 'col' | 'diagonal'
  index: number  // row/col: 0-based; diagonal: 0=main, 1=anti
}

// ClientMessage additions
type StartNewGame = { type: 'startNewGame' }

// ServerMessage additions
type WinDeclared = {
  type: 'winDeclared'
  winnerId: string
  winnerName: string
  winningLine: WinningLine
  winningCellIds: string[]
}
type GameReset = { type: 'gameReset' }

// RoomState phase union
phase: 'lobby' | 'playing' | 'ended'
```

**New shared utility:**
- `src/lib/utils/winLine.ts` — two exports:
  - `formatWinLine(line: WinningLine): string` — returns the human-readable label (`Row 3`, `Column 2`, `Top-left diagonal`, `Top-right diagonal`).
  - `winLineCellIndices(line: WinningLine, gridSize: 3 \| 4 \| 5): number[]` — returns the 0-based cell indices (row-major) on the winning line. Used by `WinLineIcon` to decide which cells to highlight.

**No new icons from lucide.** The `party-popper` icon in the winner's heading is optional and Claude's Discretion — if included, it sits immediately before the "BINGO!" wordmark in accent color. The contract is satisfied without it.
